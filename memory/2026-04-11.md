## Scope

Atlas inspection of `rainier` to establish current-state documentation and assess security, recoverability, Git-backed durability, filesystem hygiene, and layer separation.

## What was inspected

- Atlas/operator guidance from `~/.codex`
- `projects/rainier-infra` repo structure, tracked files, scripts, and docs

- host identity, OS, running systemd services, listeners, storage, timers, and top-level ownership
- AdGuard, Caddy, firewall, Route 53, and Let's Encrypt live changes made during the session

## Current-state assessment

The system is directionally sane but incomplete.

Good:

- there is already a dedicated Git-backed infra repo
- sensitive/runtime paths are mostly ignored correctly
- repo-tracked service definitions inspected in this session matched the corresponding live files
- `dns.blackridge.shumie.net` now has a publicly trusted Let's Encrypt certificate via Route 53

Weak:

- backup and restore are documented at a high level but not evidenced by verified backup jobs or backup storage
- runtime trees are mixed into `/home/shumie` with inconsistent ownership and authority boundaries
- Portainer was removed from the live host during this session
- some durable docs are stale
- host exposure is still broader than ideal because Docker-aware firewall exceptions are required and not all non-core listeners are attributed

## Findings by severity

### High

- No verified backup mechanism or backup target was identified for stateful service data or Docker-managed volumes.
- `ufw` is active, but Docker networking requires explicit bridge-subnet allows to maintain Caddy proxying and build-time DNS resolution.
- Home Assistant runtime and compose tree are root-owned under `/home/shumie`, which increases operator friction and weakens least-authority capture/sync workflows.
- AdGuard and Home Assistant expose host-network services directly, increasing the need for explicit exposure control.

### Medium

- `scripts/sync-from-live.sh` copies from live paths that are not all readable by the unprivileged repo owner in the current layout.
- `docs/HOST.md` contains stale hardware references and should be updated to match the canonical `hostnamectl` hardware identity: ZOTAC `ZBOX-CI323NANO`.
- Additional listeners were observed on `21064` and `18555` and should be attributed explicitly.
- AWS Route 53 credentials now exist as a local operational dependency for Caddy certificate management and must remain out of Git.

### Low

- No user crontab exists for `shumie`.
- No custom backup or sync timer was observed from readable systemd timer state.
- An AdGuard note already flags a likely typo in a DNS rewrite target for `claw.blackridge.shumie.net`.

## Cleanup/remediation plan

1. Decide whether to keep `ufw` inactive intentionally or enforce a host policy.
2. Define and document a canonical backup target and retention policy for:
   - Home Assistant config and database
   - AdGuard config and work data
   - Mosquitto config, password file, and persistence DB
   - Caddy runtime volumes
3. Replace brittle pull-from-live copying with a documented least-authority capture workflow.
4. Normalize service path ownership and decide which live trees should be operator-readable versus root-only.
5. Keep disposable admin tooling off the host unless it is intentionally needed and documented.
6. Refresh old docs and keep future host facts in the top-level Atlas docs.
7. Validate DoH from real client devices and document the chosen Apple rollout path.

## Recommended repo/path layout

Recommended Git repo layout:

- `ARCHITECTURE.md`
- `ENVIRONMENT.md`
- `MEMORY.md`
- `memory/YYYY-MM-DD.md`
- `compose/<service>/...`
- `services/<service>/config/` for tracked non-secret config
- `docs/restore/` for service-specific recovery procedures
- `scripts/` for capture, validation, and restore helpers

Recommended live path policy:

- `/srv/<service>/config` for durable live config
- `/srv/<service>/state` for durable runtime state
- `/srv/<service>/secrets` or equivalent root-only secret location when needed
- `/var/lib/docker/volumes/...` only for intentionally Docker-managed state
- keep Git repos out of live mutable service trees

## Backup/recovery recommendations

- Add a real backup destination with off-host copies.
- Backup stateful paths and Docker volumes on a schedule, then test restore.
- Store backup manifests and restore runbooks in Git.
- Document secret sources separately from backed-up data.
- Treat restore validation as mandatory, not theoretical.

## Clean in place or rebuild

Current recommendation: clean in place first, then reassess.

Reason:

- the tracked config is already reasonably aligned with the live service definitions
- there is no evidence yet of irrecoverable architectural sprawl
- the bigger problem is incomplete recovery design and weak layer normalization, not total chaos

Rebuild becomes the better option if:

- firewall/runtime verification reveals significant undocumented drift
- backup design cannot be added cleanly to the current path/ownership model
- other untracked live state turns out to be operationally critical and poorly understood
